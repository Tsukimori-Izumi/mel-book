# スキルクリエイター

Claude用のスキルを**対話的に**構築するガイドスキルです。  
Anthropicの「The Complete Guide to Building Skills for Claude」に基づいています。

---

## スキルとは？

スキルとは、Claudeに特定のタスクやワークフローの処理方法を教える**フォルダにパッケージされた指示セット**です。

### 3段階 Progressive Disclosure モデル

Claudeのスキルはトークンを最適化するため、以下の3段階構造で設計します：

| レベル | 内容 | ロードタイミング |
|-------|------|----------------|
| **Level 1: YAMLフロントマター** | `name` と `description` | **常時**（システムプロンプトに読込） |
| **Level 2: Markdownボディ** | メインの指示・ガイド | **関連時のみ**（descriptionで判定） |
| **Level 3: リンクファイル** | scripts/, references/, assets/ | **必要時のみ**（実行中に参照） |

> **重要**: Level 1の `description` がスキルの「門番」です。ここが不適切だとスキルがトリガーされないか、無関係なタスクで誤発火します。

---

## スキル作成ワークフロー

以下の6ステップで対話的にスキルを構築します。

---

### Step 1: ユースケース分析

**目標**: スキルが解決すべき問題を明確にする

ユーザーに以下を質問する：

1. **このスキルで何を達成したいですか？**（目的）
2. **どんな複数ステップのワークフローが必要ですか？**（手順）
3. **どんなツールが必要ですか？**（Claude組み込み or MCP）
4. **どんなドメイン知識やベストプラクティスを埋め込むべきですか？**（専門性）

回答をもとに、**2〜3個のユースケース**を以下の形式で整理する：

```
ユースケース: [名前]
トリガー: ユーザーが「[具体的なフレーズ]」と言った時
ステップ:
  1. [最初のアクション]
  2. [次のアクション]
  3. ...
結果: [期待される成果]
```

各ユースケースを以下のカテゴリに分類する：

| カテゴリ | 用途 | 例 |
|---------|------|------|
| **ドキュメント・アセット作成** | 一貫した高品質な出力の作成 | フロントエンドデザイン、文書生成 |
| **ワークフロー自動化** | 一貫した方法論による複数ステップのプロセス | スプリント計画、タスク作成 |
| **MCP強化** | MCPサーバーが提供するツールアクセスのワークフローガイド | Sentryコードレビュー |

---

### Step 2: 構造設計

**目標**: フォルダ構造と設計パターンを決定する

#### フォルダ構造テンプレート

```
[スキル名]/              ← kebab-case必須
├── SKILL.md             ← 必須：メインスキルファイル
├── scripts/             ← 任意：実行可能なコード
│   └── [スクリプト名].py
├── references/          ← 任意：参照ドキュメント
│   └── [参考資料名].md
└── assets/              ← 任意：テンプレート等
    └── [テンプレート名].md
```

#### 命名規約

| 対象 | 規則 | 正しい例 | 誤った例 |
|-----|------|---------|---------| 
| フォルダ名 | kebab-case | `notion-project-setup` | `Notion Project Setup` |
| ファイル名 | 正確に `SKILL.md` | `SKILL.md` | `skill.md`, `Skill.md` |

> **禁止事項**:
> - README.md は含めない（SKILL.md に統合する）
> - 名前に「claude」「anthropic」を含めない（予約語）

#### 設計パターンの選択

ユースケースに合わせて、`references/patterns.md` から適切なパターンを選択する：

| パターン | 使用場面 |
|---------|---------|
| **順次ワークフロー** | 特定の順序で進む複数ステップのプロセス |
| **マルチMCP連携** | 複数のサービスにまたがるワークフロー |
| **反復的改善** | 反復で品質が向上する出力 |
| **コンテキスト認識** | 同じ成果、状況によって異なるツール |
| **ドメイン特化** | ツールアクセス以上の専門知識を追加 |

---

### Step 3: フロントマター構築

**目標**: YAMLフロントマターを設計・記述する

#### 必須フィールド

```yaml
---
name: [kebab-case-のスキル名]
description: |
  [何をするかの説明]。
  「[トリガーフレーズ1]」「[トリガーフレーズ2]」「[トリガーフレーズ3]」
  と言われた時に使用する。
metadata:
  author: [作者名]
  version: 1.0.0
---
```

#### フィールドルール

- **`name`**（必須）: kebab-case、スペースや大文字なし、フォルダ名と一致
- **`description`**（必須）:
  - **何をするか** と **いつ使うか** の両方を含める
  - 1024文字以下
  - XMLタグ（`<` `>`）禁止
  - ユーザーが言いそうな具体的フレーズを含める
- **`metadata`**（任意）: author, version, mcp-server 等

#### 良い description の例

```yaml
description: |
  Figmaデザインファイルを分析し、開発者ハンドオフドキュメントを生成する。
  .figファイルがアップロードされた時、「デザインスペック」「コンポーネント文書」
  「デザインからコードへのハンドオフ」を求められた時に使用する。
```

#### 悪い description の例

```yaml
# ❌ 曖昧すぎる
description: プロジェクトを手伝う。

# ❌ トリガーが欠落
description: 洗練されたマルチページドキュメントシステムを作成する。

# ❌ 技術的すぎて、ユーザートリガーがない
description: 階層的関係を持つProjectエンティティモデルを実装する。
```

---

### Step 4: 指示記述

**目標**: スキルのメイン指示を具体的かつ実行可能に記述する

#### 指示の構造

```markdown
# [スキル名]

## 概要
[スキルの目的と価値の簡潔な説明]

## ワークフロー

### ステップ 1: [ステップ名]
[具体的で実行可能な指示]

### ステップ 2: [ステップ名]
[具体的で実行可能な指示]

## よくある問題

### エラー: [一般的なエラーメッセージ]
原因: [なぜ発生するか]
解決策: [修正方法]

## 例

### 例1: [一般的なシナリオ]
ユーザーが言うこと: 「[具体的なフレーズ]」
アクション:
1. [最初のアクション]
2. [次のアクション]
結果: [期待される成果]
```

#### 指示記述のベストプラクティス

- ✅ 良い: 「`python scripts/validate.py --input {filename}` を実行してデータ形式を確認する」
- ❌ 悪い: 「続行前にデータを検証する」
- SKILL.md はコアな指示に集中する（500行以内推奨）
- 詳細なドキュメントは `references/` に配置する
- リファレンスリンクは具体的に:「`references/api-patterns.md` を参照」

---

### Step 5: 検証

**目標**: スキルが設計通りに動作することを確認する

#### トリガーテスト

```
トリガーすべき:
  - ✅ 「[明確な関連タスク]」
  - ✅ 「[言い換えたリクエスト]」
トリガーすべきでない:
  - ❌ 「[無関係なトピック]」
  - ❌ 「[類似だが対象外のタスク]」
```

#### 機能テスト

```
テスト: [テスト名]
前提: [初期条件]
操作: スキルがワークフローを実行
期待結果:
  - [期待される出力1]
  - [期待される出力2]
  - APIエラーなし
```

#### パフォーマンス比較

| 指標 | スキルなし | スキルあり | 目標 |
|-----|----------|----------|------|
| やり取り回数 | 15回 | 2回 | 5回以下 |
| API失敗回数 | 3回 | 0回 | 0回 |
| トークン消費 | 12,000 | 6,000 | 50%削減 |

構築チェックリストは `references/checklist.md` を参照。

---

### Step 6: 改善

**目標**: フィードバックに基づきスキルを継続的に改善する

#### よくある問題と対策

| 現象 | サイン | 対策 |
|------|-------|------|
| **トリガー不足** | スキルが必要な時に読み込まれない | descriptionにキーワード追加 |
| **トリガー過多** | 無関係なクエリで読み込まれる | より具体的に条件を記述 |
| **実行品質不良** | 一貫性のない結果 | 指示の改善、エラーハンドリング追加 |

#### 改善サイクル

1. 実際の会話でスキル動作を観察する
2. ユーザーの修正操作を記録する
3. 改善事項を SKILL.md に反映する
4. `metadata.version` を更新する
